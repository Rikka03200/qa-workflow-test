# 用 Google Stitch 生成 QA-Console 前端 · 操作手册 + 分屏提示词包

> 配套文件：[`docs/DESIGN.md`](./DESIGN.md)（设计系统，先导入）。本手册告诉你**怎么喂 Stitch、按什么顺序、每屏粘什么**。
> 产物预期：在 Stitch 里得到全部页面的高保真 UI → 导出 HTML/CSS（或 Figma）→ 交回给我，对齐到我们的 HTMX + FastAPI + Jinja 技术栈并精修。

---

## 0. 为什么是「DESIGN.md + 分屏短提示词」而不是一份大简报

基于 Stitch 官方提示指南与实测，有三条**硬约束**决定了喂法：

1. **单条 prompt 不要超过 ~5000 字符**——官方确认：过长会让 Stitch**静默丢组件**。所以绝不能把所有页面写进一个大 prompt。
2. **一屏一屏来，一次只改一两处，别合并多个特性**——官方与实测（有人做数据密集的"工厂任务看板"）的共识做法是：**先生成结构骨架 → 再逐步加内容/筛选/图标**，每次一个小改动。合并需求会让它反复打乱布局。
3. **Stitch 更擅长移动端，数据密集 Web（表格/树）是它的弱项**——所以对看板、用例树、答题这种复杂屏，**降低一次成型的预期**，用"骨架→增量"的多轮提示；导出后由我来补密度与精度。

对应策略：
- **设计系统走 `DESIGN.md`**（Stitch 原生支持导入；颜色/字体/组件/明暗主题一次定死，全屏复用，避免每屏重述、也省字符）。
- **每屏一条短提示词**（下面 §4 已写好），复杂屏再跟 1–3 条**增量提示词**。

## 1. Stitch 操作流程（按这个来）

1. 打开 [stitch.withgoogle.com](https://stitch.withgoogle.com/)，新建项目。
2. **Medium 选「Web」**（不是 App）；模型可先用标准档，复杂屏不满意再切实验档（Gemini 3）。
3. **导入设计系统**：把 `docs/DESIGN.md` 的内容导入/粘贴为项目的 DESIGN.md（Stitch 支持 DESIGN.md 导入）。这一步把靛蓝主色、Inter+思源黑、等宽 JetBrains Mono、深色默认+浅色、组件圆角/徽章语义、明暗对比度全部锁定。
   - 可选本地自检：`npx @google/design.md lint docs/DESIGN.md`（Windows 用 `npx "@google/design.md" lint docs/DESIGN.md`）。出现个别 `orphaned-tokens`（focus-ring/text-faint 等）警告**无害**，那是给聚焦环/占位符留的、无对应组件属性。
4. **按 §4 顺序，一屏一屏生成**：先发该屏的「基础提示词」，出图后再依次发它的「增量提示词」。**复杂屏（看板/答题/用例树/抽检）务必单独一屏一屏做**；简单屏（登录/配置/作业抽屉）可酌情一批最多 5 屏。
5. 每屏满意后用 Stitch 的明暗切换确认**深浅两套**都成立（DESIGN.md 已给两套色）。
6. 全部完成后**导出**：优先导出**代码（HTML/CSS）**，需要的话也导出 **Figma**。把导出物（或 Stitch 项目链接 + 导出代码）发给我。

> 文案语言：下面的提示词本身就是中文，Stitch 会据此产出中文界面。若某屏中文渲染或字体不理想，可在该屏追加一句：`所有界面文案用简体中文，中文字体用思源黑体/苹方/微软雅黑`。

## 2. 通用语境前缀（可选）

DESIGN.md 导入后，风格已锁定，提示词可以很短。若某屏跑偏，在该屏提示词**前面**补这一句强化语境：

```
为企业级 WMS 测试控制台设计 Web 桌面端界面（最小宽度 1280px，信息密集，深色为默认）。沿用已导入的 DESIGN.md 设计系统：靛蓝主色，状态色绿=PASS/琥珀=WARN/红=FAIL 仅用于状态。所有文案用简体中文。
```

## 3. 屏幕清单与生成顺序

| 顺序 | 屏幕 | 复杂度 | 批量？ |
|---|---|---|---|
| 1 | 应用框架 + Sprint 看板（主屏） | 高 | 单独 |
| 2 | 工单详情（标签页框架） | 中 | 单独 |
| 3 | **答题闸门**（核心交互） | 高 | 单独 |
| 4 | **用例树视图 + 校验**（核心） | 高 | 单独 |
| 5 | 抽检结果（spot-check） | 中 | 单独 |
| 6 | 登录 | 低 | 可批量 |
| 7 | 配置（掩码） | 中 | 可批量 |
| 8 | 强模型作业进度（右侧抽屉） | 低 | 可批量 |
| 9 | 通用状态（空/加载/Toast） | 低 | 可批量 |

> 建议先做 1–5（决定整体观感的重屏），再一批做 6–9。

## 4. 分屏提示词（逐条复制；先发「基础」，再依次发「增量」）

### 屏 1 · 应用框架 + Sprint 看板

**基础：**
```
生成 Sprint 看板主页面（Web 桌面，深色）。左侧固定约 220px 窄边栏导航，自上而下：产品切换、看板、工单、知识库、配置；底部显示当前用户。顶部条：左侧面包屑「产品 / Sprint 日期」，右侧一个 Sprint 选择下拉 + 一个靛蓝主按钮「开始生成」。主区域顶部一行 4 个统计卡：候选数、运行、跳过、已覆盖。统计卡下面是一张工单决策表，列依次为：工单号、标题、角色、决策、原因、JSON校验、产物校验、测试点、待确认、操作。表头用浅色小字标签、可吸顶。整体克制、信息密集，圆角小。
```
**增量 a：**
```
在统计卡与表格之间加一条批量进度条，显示「已完成 X / 共 Y」，右侧放「续跑」「重新选单」两个次级按钮。
```
**增量 b：**
```
表格里：工单号列用等宽字体并做成可点击链接；决策列用徽章（运行=靛蓝、跳过=灰）；JSON校验/产物校验列用状态徽章（PASS=绿、FAIL=红）；待确认列当数值大于 0 时用琥珀色小徽章。行 hover 高亮。
```

### 屏 2 · 工单详情（标签页框架）

**基础：**
```
生成工单详情页（Web 桌面，深色）。顶部：工单号（等宽字体）+ 标题 + 一个状态徽章 + 右侧「返回看板」链接，最右一个小的校验状态区放两个徽章「JSON校验」「产物校验」。标题下一排标签页：需求、分析、测试点、答题、用例树、抽检、SOP，默认选中「需求」，下划线用靛蓝。内容区按 Markdown 文档渲染：一个文档标题、一个键值信息表（工单类型/状态/经办人/模块/Sprint 等）、若干分段正文。
```
**增量：**
```
把「SOP」标签页的内容做成一个带复选框的步骤清单：context、analyze、points、skeleton、detail、粘贴 CodeArts 评审、kb-extract；已完成项打勾并置灰。
```

### 屏 3 · 答题闸门（核心）

**基础：**
```
生成「待确认问题」答题页（Web 桌面，深色）。顶部标题「待确认问题清单 — EAR-246155」，下面一行说明小字，右上角一个靛蓝主按钮「保存」，按钮旁小字显示「未保存/已保存」状态。主体是垂直排列的问题卡片列表。每张卡片：顶部是「Q1：<问题陈述>」标题；下面四个只读字段，依次为 问题、来源、可能场景、影响范围，其中「可能场景」是 A/B/C 选项列表；卡片最下方是「✅ 答案」区，一个多行文本输入框。
```
**增量 a：**
```
给每张问题卡片加状态：左侧一条状态色带 + 右上角状态徽章，四种——待填（琥珀）、已自动消解（蓝）、人工已填（绿）、需人工（红）。「已自动消解」和「人工已填」的答案区改为只读文本展示（消解的额外显示来源）。只有「待填」和「需人工」的答案输入框可编辑。「需人工」卡片在答案框上方显示一行红色提示横幅「已自动检索无据，需人工/产品确认」。
```
**增量 b：**
```
在顶部标题栏「保存」左边加一个次级按钮「自动消解预览」。点击弹出一个居中对话框：左右并排显示「原文 / 归一化后」两栏差异，底部「应用」主按钮和「取消」按钮。
```
**增量 c：**
```
保存校验失败时，在页面顶部显示一条红色错误条，逐条列出未通过项；保存成功则显示一条绿色成功提示，几秒后淡出。
```

### 屏 4 · 用例树视图 + 校验（核心）

**基础：**
```
生成测试用例树审查页（Web 桌面，深色），左右两栏布局。左栏是可展开的树（思维导图风格、层级缩进、有展开箭头）：根是工单号，往下是容器节点（文件夹图标）、测试点节点、再往下是 条件/步骤/预期 三种叶子节点，三种用不同的线性图标区分。右栏是「校验问题」面板：一个可滚动列表，每条问题显示一个严重度徽章（FAIL=红 / WARN=琥珀）+ 一段等宽小字的定位路径（如 $[0].children[3].mark.priority）+ 一句问题描述。
```
**增量 a：**
```
树里：测试点节点右侧显示优先级小徽章 P1/P2/P3——P1 用靛蓝主色、P2 用中性灰、P3 用浅描边（不要用红或琥珀，避免和校验状态混淆）。存在校验问题的节点，节点行右侧加一个小圆点（FAIL 红 / WARN 琥珀）。点击右栏某条问题时，左侧对应节点高亮。
```
**增量 b：**
```
在页面顶部加一个分段切换控件「树视图 / 原始 JSON」。切到「原始 JSON」时，主区域是一个占满的等宽字体代码编辑器面板（深色代码底），底部右对齐两个按钮「校验」「保存」；当存在 FAIL 时「保存」按钮禁用置灰。
```

### 屏 5 · 抽检结果（spot-check）

**基础：**
```
生成强模型抽检结果页（Web 桌面，深色）。顶部：工单号 + 一个靛蓝主按钮「重新抽检」+ 一行概要「确认问题 N 条」。主体按 5 个维度分组展示，组标题依次为：覆盖完整性、算术与不变量、语义、越界、待确认诚信。每组下面是发现卡片列表，每张卡片含：严重度徽章（高=红 / 中=琥珀 / 低=灰）、涉及的测试点标题、问题描述、一段等宽字体的「证据」引用块、一段「建议修复」。没有问题的维度显示「无问题」占位。
```

### 屏 6 · 登录

**基础：**
```
生成登录页（Web 桌面，深色）。页面居中一张卡片（surface 色 + 细边框），卡片内：顶部产品名「QA 控制台」+ 一行副标题，用户名输入框、密码输入框，一个占满宽度的靛蓝主按钮「登录」，最下面一行小字说明「默认只读模式 · 不会写回 Jira」。背景用应用底色，极简，无插画、无装饰。
```

### 屏 7 · 配置（掩码只写）

**基础：**
```
生成配置页（Web 桌面，深色），分组卡片布局，每组卡片右上角一个「保存」按钮。第一组「凭证」：Jira 地址（普通输入框）；Jira 令牌、模型 API Key 等敏感字段一律显示为掩码状态「已配置」+ 一个「更换」次级按钮（绝不显示明文），点「更换」才出现输入框。第二组「模型」：弱模型 base_url、模型名、最大 token 三个输入。第三组「产品」：项目 Key、看板 ID、当前 Sprint 日期。第四组「安全开关」：两个开关——只读模式（默认开、用醒目样式标注）、允许写回 Jira（默认关）。
```
**增量：**
```
在页面最顶部加一条信息蓝色状态横幅：「只读模式已启用 · 不会写回 Jira」。
```

### 屏 8 · 强模型作业进度（右侧抽屉）

**基础：**
```
生成一个从右侧滑出的作业进度抽屉浮层（Web 桌面，深色，带描边和投影）。顶部：作业标题「证据消解 · 2026-06-09」+ 一个状态徽章（运行中/完成/失败）+ 右上角关闭按钮。主体是一个实时日志流区域：等宽字体、深色代码底、每行带时间戳、自动滚动到底。底部一行进度「X / Y 工单」。
```

### 屏 9 · 通用状态（空/加载/Toast）

**基础：**
```
为这个控制台生成一套通用状态组件（Web 桌面，深色）：①空状态——一个极简线性图标 + 一句说明文字 + 一个主按钮，不要插画；②加载骨架屏——表格和卡片的灰色占位条；③Toast 提示——右下角浮出，成功用绿色、失败用红色，带图标和一行文字。
```

## 5. 导出后交给我做什么

Stitch 的产物是**视觉与静态结构**，回到我这边我会负责：

- 把导出的 HTML/CSS **拆成 Jinja2 模板 + HTMX 局部**，对接 §实施文档里的 API 端点（看板、答题保存、树校验、作业 SSE 等）。
- 用 `npx @google/design.md export --format css-tailwind docs/DESIGN.md` 生成 Tailwind v4 `@theme` 变量，**统一令牌**（让 Stitch 产出的样式和我们的设计系统对齐，而不是各写一套）。
- 补足 Stitch 在**数据密集 Web** 上的短板：表格密度/吸顶、树的虚拟化与 JSONPath 内联标记、答题三态与原子保存、明暗切换、键盘可达性与 AA 对比度复核。
- 把交互态（hover/focus/disabled/loading/empty/error/409 冲突）补全并接真实数据。

> 你能给我的越多越好：**Stitch 导出的代码 / Figma 链接 / 每屏截图**任一都行；若你在 Stitch 里对某屏做了偏离 DESIGN.md 的调整，告诉我，我据此回写 `docs/DESIGN.md` 保持单一真源。

## 6. 备选：让 Stitch 从参考站"抽取"设计系统

Stitch 也支持从一个 URL 抽取设计系统。如果你心里有某个喜欢的参考站（如 Linear / GitHub / Vercel 仪表盘风），可以让它抽取后再**用我们的 `DESIGN.md` 覆盖颜色与字体**——但**主色仍坚持靛蓝、状态色仍只用于状态**，别被参考站带跑。
