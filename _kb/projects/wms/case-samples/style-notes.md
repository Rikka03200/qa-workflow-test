# WMS 用例风格注解（蒸馏自历史 4 份产物，共 237 测试点）

> 本文件是 AI 写 WMS 用例时的**风格基准**。优先级仅次于 `case-writing-spec.md`（全局硬规则）和 `codearts-json-schema.md`（JSON 契约）。

## 1. 平台标识（容器节点 text，大小写敏感）

| 平台 | 容器节点 text | 测试点标题前缀 | 范围 |
|---|---|---|---|
| WMS Web | `web` | `web_` | PC 端管理后台 |
| 仓配 App | `仓配app` | `仓配app_` | 移动端：拣货 / 收货 / 复核 / 上架 / 分拨 / 补货 / 移库 / 库存查询 / 销退收货 / 容器拆并 / 盘点 |
| 采配 App | `采配app` | `采配app_` | 移动端：直配采购录入 / 采购任务 / 收货 / 装车 / 复核 |
| 零售 App | `零售app` | `零售app_` | 乐檬零售 / WMS 直配模块对接 |
| POS | `POS` | `POS_` | POS 端（含拣货任务/收货称重打印/分拣任务等）— **大写 P/O/S** |
| TMS 司机小程序 | `TMS小程序` 或 `TMS` | `TMS_` | 物流端：运输任务 / 装车任务 / 司机交接 — **大写 TMS** |

**硬规则**：
- `POS` 和 `TMS` 是**大写**前缀（与历史用例一致），不要写成 `pos_` / `tms_`
- 多平台一个工单时：在每个平台容器下挂对应的测试点（同一功能在 web/POS/仓配app/TMS 各一份是正常的，**不要合并**）

## 2. CodeArts 容器节点层级（固定 5–6 层后挂测试点）

```
根节点（text = 用户标识+日期，如 "lzx20260414"）
└── EAR-xxxxxx                              ← 工单号（一级容器）
    └── 平台标识                            ← web / POS / 仓配app / 采配app / TMS小程序 / 零售app
        └── 业务大模块                      ← 标品管理 / 物流管理 / 设置管理 / 加工管理 / 系统设置 / 采购任务 / 拣货任务 等
            └── 业务子模块                  ← 业务查询 / 出库操作 / 入库操作 / 基础设置 / 司机交接界面 / 按门店分拣 等
                └── 具体功能页              ← ABC动态分析 / 出库订单 / 销退订单 / 新打印格式 / 物流参数 / 拣货录入 等
                    └── 测试点（mark.priority + testPoint.id + testcases）
                        └── condition / step+expect 节点
```

- **容器节点 text 用纯业务名词**：`标品管理` / `出库操作` / `销退订单`，不加前后缀
- **不要**用 `web_标品管理` / `【模块】销退订单` 这种装饰
- 工单跨多个平台时：EAR 下挂多个平台容器节点，每个平台再展开自己的模块树

## 3. 测试点标题：高频语义模板（从 237 测试点归纳）

### 3.1 字段/参数新增模板

| 场景 | 标题模板 | 示例 |
|---|---|---|
| 新增参数 | `..._增加"X"参数` | `web_物流参数_司机小程序_增加参数"退货任务生成销退任务"` |
| 参数校验 | `..."X"参数校验` | `web_ABC动态分析_筛选条件_"过滤新品"参数校验` |
| 新增列 | `..._列表_增加"X"列` 或 `..._增加"X"列` | `web_销退订单_增加"驾驶员"列` |
| 列自定义 | `..._列表_列自定义_"X"` | （见列表+排序+导出综合用例）|
| 新增字段（界面）| `..._增加"X"字段` | `仓配app_复核_复核明细_增加"整件条码1"字段` |
| 新增打印字段 | `..._增加"X"打印字段` | `web_新打印格式_..._增加"商品类别"打印字段` |
| 新增筛选条件 | `..._筛选条件_增加"X"` | `仓配app_销退收货_筛选条件_增加"驾驶员"` |
| 新增输入框 | `..._增加"X"输入框` | `POS_收货称重打印_增加"收货数量(采购单位)"输入框` |
| 新增功能 | `..._增加"X"功能` | `web_看板查询_增加"热力图"功能` |
| 新增按钮 | `..._增加"X"按钮` | `仓配app_容器拆并_容器合并_增加"全选"按钮` |

### 3.2 模板/导入模板系列

需求"模板增加 X"时，**通常拆 4 条**（视字段属性可增减）：

```
web_出库订单_批量导入线下订单_模板增加"有效日期"
web_出库订单_批量导入线下订单_"有效日期"默认校验      ← 不填走默认值
web_出库订单_批量导入线下订单_"有效日期"取最迟        ← 多明细合并规则
web_出库订单_批量导入线下订单_"有效日期"格式校验      ← 格式错误处理
```

### 3.3 参数应用 / 开关组合

| 场景 | 模板 |
|---|---|
| 参数 ON 行为 | `..._应用参数"X"=true` |
| 参数 OFF 行为 | `..._应用参数"X"=false` |
| 开启行为 | `..._开启"X"_Y校验` |
| 关闭行为 | `..._关闭"X"_Y校验` |
| 按 X = true | `..._按X装车=true_"Y"取值` |
| 按 X = false | `..._按X装车=false_"Y"取值` |

示例：
- `TMS_运输任务_司机交接_退货明细tab_应用参数"退货任务生成销退任务"=true`
- `TMS_运输任务_司机交接_退货明细tab_应用参数"退货任务生成销退任务"=false`
- `web_装车单_明细_按托盘装车=true_"集货位"取值`
- `web_装车单_明细_按托盘装车=false_"集货位"取值`
- `web_波次策略_开启"启用超量补货"_生成超量补货校验`
- `web_波次策略_关闭"启用超量补货"_生成超量补货校验`
- `零售app_货运公司_开启WMS_权限校验`
- `零售app_货运公司_关闭WMS_权限校验`

### 3.4 边界值 / 数值比较条件

| 场景 | 模板 |
|---|---|
| 大于 | `..._X>Y_..._行为` |
| 小于 | `..._X<Y_..._行为` |
| 等于 | `..._X=Y_..._行为` |
| 业务差异点 | `..._订货量小于采购量` / `..._多批次累计不超采购量` |

示例：
- `采配app_采购任务_直配采购录入_订货总量>采购量_按订货比例分配`
- `采配app_采购任务_直配采购录入_分批次分配_按订货比例分配`
- `web_热力图_查询维度=出库频率_库位颜色展示`
- `web_热力图_查询维度=出库数量_库位颜色展示`

### 3.5 权限校验

固定写法：`..._权限校验`，作为单独测试点，覆盖"开/关权限位 → 入口可见性 / 操作可用性"。

示例：
- `web_收货任务_收货看板_权限校验`
- `web_送货预约_预约看板_权限校验`
- `web_热力图_权限校验`
- `零售app_货运公司_开启WMS_权限校验`

### 3.6 交互优化

`..._XX交互优化` 用在 UI 行为变化（不是字段新增），常见于点击/跳转/确认弹窗。

示例：
- `POS_拣货任务（生鲜）_按门店分拣_拣货录入_"拣货商品"交互优化`
- `web_装车单_明细_按托盘装车交互优化`
- `POS_拣货任务（生鲜）_按门店明细拣货_拣货录入_点击"确定"交互优化`

### 3.7 跨平台联动

多平台一个功能时，每个平台一条标题，**结尾保持业务功能点一致**：

```
web_销退订单_增加"驾驶员"列                          ← Web 列表
仓配app_销退收货_筛选条件_增加"驾驶员"               ← App 筛选
仓配app_销退收货_列表_增加"驾驶员"                   ← App 列表
TMS_运输任务_司机交接_退货明细tab_应用参数"..."=true  ← TMS 联动
```

## 4. 标题硬规则（违反即修订）

1. **末段必须是业务功能点**——读用例的人没看过工单也能猜到测什么
2. **禁止占位词**：`场景1` / `场景N` / `样例N` / `case_N` / `第一种` / `回归_EAR-xxxxxx` / `bug_xxxxxx`
3. **禁止在标题里嵌工单号**：工单号放在 testPoint.id 或前置条件溯源里，不进标题
4. **平台前缀大小写必须严格匹配**：`POS` 不是 `pos`；`TMS` 不是 `tms`；`web/仓配app/采配app/零售app` 全小写
5. **业务关键值用中文双引号包裹**：`"有效日期"`、`"过滤新品"`、`"驾驶员"`、`"按订货比例分配"`——不用英文引号 `""`、不用 markdown 反引号、不用 `<b>` 加粗
6. **多场景拆分时**：用业务差异点命名（`订货量小于采购量` / `订货量等于采购量` / `订货量大于采购量`），而不是"场景 1/2/3"
7. **同一个改动同时影响多平台时**：各平台一条独立测试点，**前提是两端实际行为有差异**；若两端行为完全相同可只测一端

## 4.1 必要充分原则（**核心约束**——优先级高于本文件 §3 的标题模板）

**用例数应反映"修改方案"实际新增的功能切面，而不是机械套用标题模板**。

| 工单难度 | 经验用例数 | 典型范围 |
|---|---|---|
| 1 星（小参数 / 小字段 / 单分支）| **3–6** | 入口可见性 + 主路径 + 边界 |
| 2 星（参数+联动+多分支 / 单功能多场景）| 5–10 | + 联动场景 + 异常路径 |
| 3 星（新模块 / 跨平台联动 / 复杂规则）| 10–20 | + 兼容性 + 权限 + 各端覆盖 |
| 4 星+（大方案 / 全流程改造）| 20+ | 全维度 |

**用例切分必要性自检**（每条用例写下来前问自己）：

| 自检项 | 通过条件 |
|---|---|
| 该用例对应方案里**写明的**哪一行？| 指得出来 |
| 拆 ON/OFF 两条是因为**两条分支行为差异显著**？还是只是机械套模板？| 必须前者 |
| 拆"等于 / 大于 / 小于"是因为方案**显式说**有边界？还是 AI 自己想的覆盖维度？| 必须前者 |
| 把校验组合到主路径用例的 step 序列里，会不会同样测全？| 能合并就**合并** |
| 这条用例缺了，方案的功能切面是否还测全？| 缺了真的少覆盖 |

**禁止凑数的"覆盖维度模板"**——以下场景**默认不写**测试点，除非方案明确提到：

- 必填校验 / 长度校验 / 字符集校验（除非方案说"校验 XX 字段格式"）
- 重复值校验（除非方案说"保存重复校验"）
- 排序 / 导出 / 列自定义（除非方案说"列表新增 X 列支持排序/导出"）
- 权限校验（除非方案说"新增 XX 权限位"）
- 修改记录（除非方案说"修改记录保存 XX 字段"）
- UI 文案校验、注释校验、提示语校验（**不拆独立用例**；UI/提示文案用功能性描述直接写进主路径 expect、**不标 `[待确认]`**，不纠结精确措辞——详见 `_kb/_global/case-writing-spec.md §5`。只有提示的确切措辞本身就是要校验的规则点、且方案给出原文时才精确引用原文）

**方案规则 → 用例 1:1 映射强制要求**：

写第一个用例前，AI 必须先列出方案的"规则切面清单"（按 1/2/3 编号），然后**每个用例必须能指向清单中的某一条**。例如 EAR-240452 修改方案的切面清单：

```
切面 1：参数定义 + 默认值 + 联动置灰      → A1 一条
切面 2：开启 + 早于 + 存储位≠分拣位 → 过滤   → A2 一条
切面 3：开启 + 例外（存储位=分拣位）→ 不过滤 → A3 一条
切面 4：关闭 + 早于 → 不过滤              → A4 一条
```

每个用例的 condition 必须**精确反映**所对应切面的语义（变量取值要把切面的关键判断条件构造出来）。

**方案语义逐字理解**（CLAUDE.md §3.3 第 8 条加固）：

方案描述的主语 / 条件 / 比较对象**不能被简化、类比、或自动补全**：

- 方案"X、Y 生产日期相同" → 主语就是 X 和 Y，不是 X 和 Z；不要自动类比成"更主流的语义"
- 方案"X ≥ Y" → 严格写 ≥；不许理解为"等于"或"晚于"的简称
- 方案"判断提示后，在 X 时校验" → 校验触发点是 X 一个点，"判断提示后"只是时序描述
- 拿不准时回去再读 5 遍方案原文 + 与流程图交叉验证

**参数 ON/OFF 拆分**：
- 方案两条分支**行为差异显著**（如开启=强制路径 A、关闭=允许路径 B）→ 拆 2 条
- 方案关闭分支只是"不生效，走原有流程" → 可只测开启分支（关闭分支用 1 个 step 顺带验证）
- 方案两条分支差异写得很具体——拆 2 条不要省

**跨平台拆分**：
- 两端是同一公共页/同一参数应用——可只测一端（在前置条件标"web 端 / 仓配 App 端均适用"）
- 两端入口/交互/字段确有差异——拆 2 条

**示例（EAR-240452 反面教材→正面修订）**：

| 反面（18 条）| 正面（4-5 条）|
|---|---|
| A1 参数新增 + A2 修改保存 + A3/A4 联动开关 + A5 切换联动 = 5 条 | 合并为 **1 条**：参数入口 + 默认值 + 联动置灰 + 保存 用同一测试点的多 step 覆盖 |
| B1/B2/B3/B4 开启 4 分支 | 保留 **2 条**：早于 + 等于（方案显式说差异）；晚于/无库存可不写（方案没说差异/没说边界）|
| C1/C2 关闭 2 分支 | 保留 **1 条**：早于（验证走 EAR-105519 旧流程）|
| D1/D2/D3/D4 公共页双端 | **删除整组**——公共页本身就是上架步骤的一环，已含在 B/C 用例的 step 中 |
| E1/E2 历史规则兼容 | **删除**——方案没要求测兼容性 |
| F1 权限校验 | **删除**——方案没说新权限 |

## 5. condition / step / expect 写法蒸馏

### 5.1 condition（前置条件）

- **单条**：不编号，直接一句话。例：`门店A，拣货任务有商品1/2/3/4/5/6/7`
- **多条**：用 `<br>` 换行，从第二条开始编号 `2.` `3.`（**第一条通常不写 "1."**）。例：
  ```
  商品A，批次商品，采购属性=市场自采<br>商品B，非批次商品，采购属性=市场自采<br>商品C，批次商品，采购属性≠市场自采<br>2.以上商品均新增采购任务
  ```
- **顺序约定**：参数配置 → 档案数据 → 库存批次 → 单据准备 → 操作前的状态
- **大量配置时**也可全编号（包括第一条 `1.`），与第二条以后混合编号也常见——两种写法都接受，关键是清晰

### 5.2 step（步骤）

- **极简模板**：`{页面路径}，{动作}`，多个动作用中文逗号串联
- **页面路径**用 `-` 连接：`wms-出库订单-批量导入线下订单` / `TMS司机小程序-运输任务-司机交接-退货明细tab`
- **动作动词化**：`下载模板` / `填写必填项` / `导入` / `点击保存` / `查看修改记录` / `开启"X"`
- **不要**写"测试 X 功能" / "进入页面看看" / 缺位置缺动作

示例（来自实际产物）：
- `wms-ABC动态分析-筛选条件`（极简——只是描述当前位置）
- `wms-出库订单-批量导入线下订单，下载模板，不填写"有效日期"，填写必填项，导入`
- `修改"有效日期"为当天+1，导入`
- `档案-角色管理-编辑-权限管理，开启：WMS-标准管理-业务规则-批次推荐规则-编辑`

### 5.3 expect（预期）

- **极简、可校验**：字段=值 / 显示"弹窗" / 状态=值 / 校验通过
- **关键文案/字段用中文双引号包裹**：`显示"过滤新品"参数`、`显示"驾驶员"列`
- **不要**写 `成功` / `页面正常` / `应该可以`

示例（来自实际产物）：
- `显示"过滤新品"参数，默认不勾选`
- `导入成功，单据"有效日期"为当天`
- `导入成功，客户A单据有效日期为"当天+10"`
- `导入失败，显示格式错误提示`
- `保存成功，回显正确`
- `列表显示3条明细，分别显示托盘1，集货位=JH01；托盘2，集货位=JH01；托盘3，集货位=JH02`
- `批次属性页面为空`
- `跳转至商品3拣货录入页面，列表显示商品4/5/6/7，可上划看到商品1/2`

## 6. 一个测试点 = 多少 step + expect？

观察 237 测试点的拆分粒度：

| 类型 | 节点数 | 例 |
|---|---|---|
| **最简**（无前置）| 1 step + 1 expect | 列表/筛选条件可见性验证 |
| **标准**（含前置）| 1 condition + 1–3 step+expect | 字段新增/参数行为 |
| **复杂联动** | 1 condition + 4–6 step+expect | 跨平台流程、保存+回显+修改记录组合 |
| **覆盖全字段**（导出+排序+导出+保存）| 1 condition + 4 step+expect | 列新增 → 自定义勾选 → 排序 → 导出 |

**经验**：
- 单测试点 step 一般 **1–6 个**，超过 6 个考虑拆分两条测试点
- 每个 step 必须**恰好 1 个 expect** 子节点（JSON 契约硬要求）
- 前置条件**至多 1 个 condition 节点**——多条前置写在同一节点 text 里，用 `<br>` 分隔

---

## 7. 优先级判定（基于实际分布）

| 等级 | 出现频率 | 适用 |
|---|---|---|
| P1 | ~5% | 新参数主入口、核心新增功能（如 EAR-240267 的 `web_物流参数_司机小程序_增加参数"退货任务生成销退任务"` 是 P1）|
| P2 | ~95% | **默认**——所有常规字段新增 / 列新增 / 校验 / 联动 / 跨平台同步 |
| P3 | 0%（历史用例未使用）| 边缘 UI / 视觉细节 |

**P1 判定线索**（出现其一即可考虑 P1）：
- 新模块的总开关参数（开启后改变全局行为）
- 新功能的"新增/入口"测试点（区别于"校验"系列）
- 涉及阻塞性流程（保存校验、必填、唯一性）

## 8. 加粗 / 换行 / HTML 实体

- **多条前置**：用 `<br>` 换行（CodeArts 实测可渲染）
- **加粗**：**禁用** `<b>` / `<strong>` / `**...**` —— CodeArts 不渲染，按字面显示
- **强调关键值**：用中文双引号 `"..."` 框选
- **不要**用 `《span...》` 损坏 HTML 标签
- **不要**用 `&lt;br&gt;` / `&quot;` 等 HTML 实体编码
- **不要**在 JSON 节点 text 里嵌入 markdown 反引号 `` ` ``
