# 用例编写规范（跨产品通用）

> 本文件适用于**所有产品线**。产品特定的规则（如术语、模块路径）见 `_kb/projects/<product>/`。

---

## 1. 测试点标题命名

格式：`{平台}_{模块路径}_{测试要点}`

| 元素 | 取值 |
|---|---|
| 平台 | **产品级固定**（在产品 README / style-notes 列举）；大小写敏感——例如 WMS 项目 `web` / `仓配app` / `采配app` / `零售app` 用小写，但 `POS` / `TMS` 用大写 |
| 模块路径 | 业务模块，下划线连接，从粗到细 |
| 测试要点 | 简短描述本用例验证什么，关键字段/参数名用**中文双引号** `“...”` 包裹 |

示例：
- ✅ `web_批次推荐规则_列表_删除校验`
- ✅ `web_商品档案_WMS商品_增加“批次推荐规则”`
- ✅ `仓配app_拣货_不锁库流程_推荐批次优先级_拆零需求小于一件`
- ✅ `仓配app_容器拆并_容器合并_扫描条码定位商品`
- ✅ `POS_拣货任务（生鲜）_按门店分拣_拣货录入_“拣货商品”交互优化`
- ✅ `TMS_运输任务_司机交接_退货明细tab_应用参数“退货任务生成销退任务”=true`
- ❌ `测试删除按钮`（无平台、无路径、不动词化）
- ❌ `web_删除`（路径太浅）
- ❌ `WEB_批次推荐规则_列表_删除`（平台标识大小写错误）
- ❌ `pos_拣货任务_..._交互优化`（POS 应大写）
- ❌ `仓配app_拣货_不锁库流程_场景1_拆零8包`（**禁止**用"场景 N"做标识——读用例的人看不出测的是什么）
- ❌ `采配app_采购任务_直配商品_不超采购量_EAR-249140回归`（**禁止**在标题里嵌工单号——标题写功能点，工单号放 testPoint 备注或前置条件里）

**标题硬约束**：
1. 末尾必须是**业务可读的功能点**，让没看过工单的人能立刻知道在测什么
2. **禁止**：`场景1`、`场景 N`、`样例N`、`example_N`、`case_N`、`回归_EAR-xxxxxx`、`bug_xxxxxx`、`第一种`、`第二种` 等无信息量的占位
3. 多场景时用业务差异点命名：`订货量小于采购量` / `订货量等于采购量` / `订货量大于采购量` / `分批次累计不超采购量` 等
4. **关键字段/参数名/选项值/按钮文案** 一律用**中文双引号** `“...”` 包裹（不要用英文双引号、markdown 反引号、`<b>` 加粗）
5. **平台前缀大小写**严格按产品 README/style-notes 约定，不要自创变体
6. 跨平台同一改动时**不要合并**——每个平台一条独立测试点

## 2. 容器节点 text

容器节点（非测试点）直接用业务名词，不加前后缀：
- ✅ `标准管理`、`业务规则`、`批次推荐规则`、`商品档案`
- ❌ `web_标准管理`、`标准管理模块`、`【模块】标准管理`

## 3. 前置条件写法

前置条件节点（`condition: "Y"`）：

- 多条用编号 `1.`、`2.`、`3.`，**用 `<br>` 换行**（CodeArts 实测可渲染 `<br>` 换行）
- **`<b>`、`<strong>` 严禁使用**——当前 CodeArts 实例不渲染加粗标签，会按字面显示原文。需要强调时直接写“参数=值”，让上下文表达，或用中文双引号 `“...”` 框选关键值
- **字段名/参数名/选项值/按钮文案**在所有用例正文中统一使用中文双引号 `“...”`；禁止使用 markdown 反引号 `` `...` `` 或英文双引号。
- **包含全部必要的**：参数配置、数据准备、用户身份、初始状态

示例：
```
1.wms-标准参数，“仅允许领取生产日期最早商品”=true；“允许跨分组领取拣货任务”=按分组；<br>
2.A库区库位：A-01，B库区库位：B-01；<br>
3.按库位分拣分组：<br>
A组库位:A-01（仅商品A库存：20260101 批次/100公斤），拣货员：用户A；<br>
B组库位:B-01（仅商品A库存：20250101 批次/100公斤），拣货员：用户B；<br>
4.波次策略，分拣方式=按单分拣，优先级=“分拣分组>生产日期>库位库存”；<br>
5.新增2张出库订单：客户A添加商品A数量=100；客户B添加商品A数量=100；生成波次计划单
```

## 4. 步骤写法

步骤节点（`step: "Y"`）：

**模板**：`{产品/页面短路径}，{动作描述}`

- 动作必须动词化：点击、输入、选择、勾选、上传、扫描、登录、切换
- 步骤里的页面定位使用**业务入口短路径**，不要写 CodeArts 用例存放目录全链路
- WMS web 用例优先写成 `wms-{页面}` 或 `wms-{页面}-{列表/新增/编辑/明细/tab}`，例如 `wms-出库单`、`wms-出库单-列表`、`wms-出库单-明细`
- `web > 标品管理 > 出库操作 > 出库单` 这类路径只用于 JSON 容器节点，不进入步骤正文
- 多个连续小动作可合并为一个步骤（保持语义完整）

示例：
- ✅ `wms-出库单，点击“批量出库”，导入已填写“基本数量”的浮动规格批次商品明细`
- ✅ `wms-出库单-明细，查看正式出库单明细`
- ✅ `wms-批次推荐规则，未勾选规则`
- ✅ `登录用户B仓配App-拣货，点击领取`
- ✅ `档案-角色管理-编辑-权限管理，开启：WMS-标准管理-业务规则-批次推荐规则-编辑`
- ✅ `商品档案-wms商品-批量修改，勾选：分拣分组：分组2、3，保存`
- ❌ `wms-标品管理-出库操作-出库单，点击“批量出库”`（把 CodeArts 存放路径写进步骤）
- ❌ `点击按钮`（缺位置、缺哪个按钮）
- ❌ `进入页面看看`（不动词化）
- ❌ `测试登录功能`（这是测试点不是步骤）

## 5. 预期写法

预期节点（`expect: "Y"`）：

**必须可校验**：包含字段名 + 值 + 文案 + 页面/状态变化。

示例：
- ✅ `领取成功，拣货录入显示分拣库位=B-01，生产日期=2025-01-01，批次可拣量=100公斤`
- ✅ `显示“批量删除”弹窗`
- ✅ `必填项，支持最多20个字符，仅允许输入英文、数字`
- ✅ `导入成功，商品详情中回显正确，修改记录标记成功`
- ✅ `保存按钮置灰，不可点击`
- ❌ `成功`（无法校验）
- ❌ `页面显示正常`（不明确）
- ❌ `应该可以登录`（不可执行）

**UI 元素文案（按钮 / 弹窗 / 标题 / 菜单项）**：用**功能性描述直接写**，不必纠结精确文案、**不要标 `[待确认]`**——产品常常没定这些文案、最终由开发定，执行者看得懂即可。例：`点击“打印”` 或 `点击打印` 均可；`显示“预生成集货托盘”弹窗`（不必纠结弹窗标题确切叫什么）；`弹出个数输入弹窗`。

**业务 / 规则性提示语**：只有当某条提示的**确切措辞本身就是用例要校验的规则点**（如校验失败、状态拦截的具体话术）、且方案给出了原文时，才精确引用原文（如 `未找到托盘号！`）。方案未给原文的，**用功能性描述即可**（如 `提示托盘号无效，无法继续`），**不必**单独标 `[待确认]` 去等文案。

> `[待确认]` 只留给**业务行为 / 规则本身**不确定的地方（见 §8），**不要**用来等 UI 文案。

## 6. 优先级判定

| 等级 | 标记 | 适用 |
|---|---|---|
| P1 | `{"mark":{"priority":{"1":true}}}` | 新功能主入口、核心新增字段、阻塞性流程、关键权限 |
| P2 | `{"mark":{"priority":{"2":true}}}` | **默认**；常规功能/边界/校验/异常/权限/兼容 |
| P3 | `{"mark":{"priority":{"3":true}}}` | 边缘场景、UI 视觉、低概率组合 |

经验值：单个工单 P1 一般不超过总数 20%。

## 7. 覆盖维度检查表

新模块/新字段时，AI 必须覆盖：

- [ ] **入口可见性**：模块/字段在何处显示，是否符合权限
- [ ] **必填校验**：必填字段为空时的提示
- [ ] **长度校验**：超长输入、刚好边界、最短输入
- [ ] **字符集校验**：英文/数字/中文/特殊字符/空格
- [ ] **唯一性校验**：重复值的提示
- [ ] **默认值**：新建/编辑时默认显示
- [ ] **联动**：与其他字段的依赖、置灰条件
- [ ] **保存/重置**：保存成功后的状态、回显、修改记录
- [ ] **权限矩阵**：查询/编辑/删除/导入/导出 各权限位独立测试
- [ ] **批量操作**：勾选/全选/全不选、批量校验
- [ ] **导入导出**：模板、必填校验、错误数据、回显
- [ ] **历史功能联动**：与已有参数/规则的组合（开/关组合都要覆盖）

---

## 8. 抗幻觉硬规则

1. 文档没写的 → `[待确认: ...]` 标记
2. 引用规则 → 标章节号或来源
3. 不要"看起来应该这样"地补字段
4. 不要"通常这种场景..."地补流程
5. 在产出末尾汇总"待确认问题清单"

## 9. 反例与修正样本

### 反例 1：步骤太粗
- ❌ `测试新增功能`
- ✅ `wms-批次推荐规则-新增，规则代码=001，规则名称=规则1，启用=默认勾选，点击保存`

### 反例 2：预期不可校验
- ❌ `保存成功`
- ✅ `保存成功，列表新增一行：规则代码=001，规则名称=规则1，启用状态=是，备注=空`

### 反例 3：编造文案
- 需求：「保存时校验代码唯一性」
- ❌ 预期写 `提示"规则代码已存在，请修改"`（这个确切文案需求没写，属编造）
- ✅ 预期写 `触发重复校验，提示代码重复、保存被拦截`（功能性描述，不纠结确切措辞、**不标 `[待确认]`**）

### 反例 4：跨规则猜测
- 需求只说"新增'低优先级'字段"
- ❌ 测试点扩展到"低优先级与跨波次的组合"（需求没说会与跨波次互斥）
- ✅ 测试点只覆盖"低优先级勾选/取消勾选"行为 + 标 `[待确认: 低优先级是否与跨波次互斥]`

### 反例 5：标题用"场景 N"或工单号
- ❌ `采配app_采购任务_直配商品_按订货比例分配_场景1_订货小于采购`
- ❌ `采配app_采购任务_直配商品_按订货比例分配_不超采购量_EAR-249140回归`
- ✅ `采配app_采购任务_直配商品_按订货比例分配_订货量小于采购量`
- ✅ `采配app_采购任务_直配商品_按订货比例分配_多批次累计不超采购量`
- 理由：读用例的人没看过工单，看"场景 1"猜不出测的是什么；看"EAR-xxx 回归"还得跳去查工单。标题要让功能点自解释。

### 反例 6：用 `<b>` 加粗
- ❌ 步骤 text：`点击<b>按订货比例分配</b>` （CodeArts 不渲染，会按字面显示 `<b>`）
- ✅ 步骤 text：`点击“按订货比例分配”` 或 `点击 按订货比例分配 按钮`
