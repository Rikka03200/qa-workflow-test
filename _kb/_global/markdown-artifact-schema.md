# Markdown 产物契约

> 本契约约束 `tickets/<product>/<sprint-date>/<ticket>/` 下除 `test-design.json` 之外的工程产物。目标是让每一步产物可被人审、可被后续 slash 命令稳定读取，并避免把对话报告混入文件正文。

---

## 1. 通用规则

### 1.1 文件角色

| 文件 | 角色 | 生成阶段 |
|---|---|---|
| `README.md` | 工单索引、SOP 进度、人工决策记录 | `/qa:new` 初始化，后续可追加 |
| `requirement.md` | Jira 原始信息与 L1/L2/L3 分级后的需求真源 | `/qa:new` |
| `linked-issues.md` | 直接关联工单与参考工单全文摘录 | `/qa:context` |
| `business-context.md` | 当前工单可引用的业务上下文与证据链 | `/qa:context` |
| `analysis.md` | 需求理解、风险判断、待确认问题、测试范围 | `/qa:analyze` |
| `questions.md` | 给用户填写的待确认问题表单 | `/qa:analyze`，无问题时可不生成 |
| `test-points.md` | 已拆解的测试点标题、优先级、覆盖依据 | `/qa:points` |

### 1.2 文件正文禁止内容

以下内容只能出现在聊天回复中，不能写入 Markdown 产物正文：

- 完成后报告：`✓ 已生成`、`请审增删`、`下一步运行 /qa:...`、`人工动作` 等。
- AI 执行过程流水：`我已读取`、`我认为`、`本次生成了`、`输出自查报告` 等。
- 未完成占位：`待执行于 analyze 补充`、`TODO`、`待补`、`稍后补充`。确实未知时必须写成 `[待确认: 具体问题]`，并进入 `analysis.md` / `questions.md`。
- CodeArts 不支持的正文格式：`<b>`、`</b>`、`<strong>`、`</strong>`、`《span...》`、`&lt;`、`&gt;`、`&quot;`。

### 1.3 结构规则

- 每个文件只允许 1 个一级标题 `#`，且必须在第 1 行。
- Jira wiki 中的 `# 客户场景`、`# 修改方案` 等原始标题必须降级到当前章节下，不能保留为 Markdown 一级标题。
- 固定章节没有内容时写 `无`，不要留空或写“待回填”。
- 来源、JQL、关联工单证据允许写在 Markdown 产物中；但不得进入 `test-design.json` 的 `condition` / `step` / `expect` 节点。
- 业务字段名、按钮、参数、选项值优先用中文双引号 `“...”`，不要用 markdown 反引号表示 CodeArts 正文样式。

## 2. 目录齐套要求

完整工单目录在 `/qa:detail` 后应至少包含：

```text
README.md
requirement.md
linked-issues.md
business-context.md
analysis.md
questions.md        # 无待确认问题时可不存在，但 analysis.md 必须声明“无”
test-points.md
test-design.json
attachments/        # 空目录占位；只记录元数据，不下载附件
```

允许额外材料文件，例如专项分析、流程图转写、接口字段对照，但必须满足：

- 文件名能说明用途。
- `README.md` 的“关键决策记录”或“附加材料”中登记。
- 不作为默认读取入口，除非用户或当前步骤明确引用。

```markdown
## 7. 用例编写约束

- 命名：`{平台}_{模块路径}_{测试要点}`。
- 前置条件：编号 + `<br>` 换行；禁止 `<b>` / `<strong>`。
- 步骤：短业务入口路径 + 动作，不写 CodeArts 容器全链路。
- 预期：字段、值、状态可校验；提示/UI 文案用功能性描述，不纠结确切措辞、不标 `[待确认]`（见 `_kb/_global/case-writing-spec.md §5`）。未知的业务行为/规则才标 `[待确认: 简述]`。
- JSON：单根树、32 位大写十六进制 UUID、condition 第一位、每个 step 恰好 1 个 expect。
```

## 8. `questions.md`

`questions.md` 是唯一人工答题入口，格式必须稳定、可机读。

### 8.1 无待确认问题

```markdown
# 待确认问题清单 — <ticket>

无
```

标题后正文必须且只能写一行 `无`；不得同时出现 `## Q` 题块。

### 8.2 有待确认问题

```markdown
# 待确认问题清单 — <ticket>

> 请在每个“✅ 答案”下方填写确认结果；如果暂时无法确认，填写 `[待确认]`。

## Q1: <问题陈述>

**问题**：
<问题详细描述>

**来源**：
<requirement.md / business-context.md / linked-issues.md / _jira-search.md / rules.md 中的来源>

**可能场景**：
- A. ...
- B. ...

**影响范围**：
<不同答案会导致哪些测试点、步骤或预期变化。>

**✅ 答案**：
<!-- 请在下方填写，删除本行注释后填写 -->
```

硬规则：
- 第 1 行必须固定为 `# 待确认问题清单 — <ticket>`。
- 题块必须使用连续的 `## QN: <问题陈述>`，`N` 从 1 连续递增。
- 不允许额外二级分节；证据完整性审查补充问题也直接追加为下一个 `## QN: ...` 题块。
- 每题必须且只能按顺序包含五个字段：`**问题**：`、`**来源**：`、`**可能场景**：`、`**影响范围**：`、`**✅ 答案**：`。
- `问题`、`来源`、`可能场景`、`影响范围` 必须有非空正文；HTML 注释不算正文。
- `✅ 答案` 是唯一人工/自动答案入口；未填时保留 HTML 注释占位，自动消解答案需写明来源。
- 该格式由 `scripts/validate-questions.py` 强校验；不稳定格式不得进入后续 `points/design` 生成。
