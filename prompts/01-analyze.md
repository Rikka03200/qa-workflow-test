# /qa:analyze — 需求理解与风险识别

> 用法：`/qa:analyze`
> 前置：`/qa:new` 已执行；建议先跑 `/qa:context` 让上下文更完整。

---

## AI 任务

阅读：

- `tickets/<product>/<sprint-date>/<ticket>/requirement.md`
- `tickets/<product>/<sprint-date>/<ticket>/business-context.md`（如存在）
- `tickets/<product>/<sprint-date>/<ticket>/linked-issues.md`（如存在）
- `_kb/_global/markdown-artifact-schema.md`
- `_kb/_global/qa-methodology.md`
- 必要时 MCP 拉 Jira 关联工单

本步骤只输出分析产物，不生成测试点列表；测试点列表属于 `/qa:points`。

---

## 1. 写入 `analysis.md`

严格按 `_kb/_global/markdown-artifact-schema.md §7` 写入：

```markdown
# 需求分析 — <ticket>

## 1. 功能清单
## 2. 方案切面清单
## 3. 风险与影响分析
## 4. 待确认问题清单
## 5. 测试范围
## 6. 测试点数量预估
```

### 1.1 功能清单

按“模块 / 子模块 / 功能点”分层，每条标注：

- `[新增]` / `[修改]` / `[删除]`
- 影响的已有模块 / 参数
- 来源（只允许 L1 方案、关联工单 L1 或已确认用户回答）

功能清单只能来源于 L1 明确条目，不要补“通常这种模块应该有的”。

### 1.2 方案切面清单

用表格列出每个 L1 切面，作为 `/qa:points` 的 1:1 映射依据：

| # | L1 切面 | 来源 | 后续覆盖建议 |
|---|---|---|---|
| 1 | ... | ... | P1/P2，建议测试点方向 |

要求：

- 每个切面必须能指向 `requirement.md ## 3`、`linked-issues.md` 中 L1 段，或用户已确认答案。
- L2/L3 只能写入风险、边界或待确认问题，不得作为功能切面。
- 存疑时降级，不要把背景改写成规则。

### 1.3 风险与影响分析

必须按 `_kb/_global/qa-methodology.md §6` 的 9 问输出完整表格：

| 风险维度 | 是否触及 | 说明 |
|---|---|---|
| 1. 影响范围 | 是/否/不确定 | ... |
| 2. 不可逆变更（数据迁移） | 是/否/不确定 | ... |
| 3. 核心算法变更 | 是/否/不确定 | ... |
| 4. 接口契约变更 | 是/否/不确定 | ... |
| 5. 权限模型变更 | 是/否/不确定 | ... |
| 6. 大数据量场景 | 是/否/不确定 | ... |
| 7. 并发场景 | 是/否/不确定 | ... |
| 8. 离线/弱网（App 类） | 是/否/不确定 | ... |
| 9. 历史数据兼容 | 是/否/不确定 | ... |

末尾给总体风险评估：高 / 中 / 低 + 主要理由。

### 1.4 待确认问题清单

任何 `requirement.md` + `business-context.md` + `linked-issues.md` 中不清晰、有歧义、与历史规则冲突的点，必须列出。

格式：

```markdown
### Q1: <问题陈述>

- 来源：<逐条列出与本问题相关的**真实、可核对、能按图索骥找到**的来源，每条都点名出处并带原句（加引号）：
  · 本工单方案 → `本工单修改方案：“<原句>”`
  · 本工单评论 → `本工单 <评论人> <日期> 评论：“<原句>”`
  · 关联工单评论/方案 → `<工单号> <评论人> <日期> 评论：“<原句>”`（例：`EAR-252916 陈嘉豪 2026-06-03 评论：“接口功能为查询批发客户信息，支持传入条件‘客户代码’”`）
  · 知识库规则 → `业务规则 <§章节>：“<原句>”`
  **禁止**：①写“背景信息/关联讨论/相关讨论/有提到/资料显示”这类无主语、无出处的笼统标签（引用评论必须点名「工单号+评论人+日期」）；②把 **L3 背景**当依据——`requirement.md §2 客户与背景`/`背景与价值`/`客户场景`/`期望目标`/`功能详情` 等只是背景上下文，**不是修改方案/业务规则**，绝不可作为来源。无任何可核对的 L1 来源时写“无可确认来源”。>
- 已检索：<JQL 或“无业务规则类疑问，不适用 JQL”>
- 候选答案：A / B / C
- 影响：<不同答案会导致测试范围、步骤或预期怎么变>
```

问题陈述要尽量用白话，避免“口径”等内部术语；优先写成“本次是否需要覆盖...”“最终规则是什么...”“验收范围是否包含...”这类直接问题。

**消解优先（列入待确认前必做）**：任何业务规则/算法/参数行为/字段联动疑问，列入待确认前必须先在以下证据里逐一找答案——`linked-issues.md`（关联工单描述+评论）、`business-context.md §3/§5`（`rules.md` 摘录 + Jira 搜索记录）、`_jira-search.md`（命中工单全文）、同机制历史规则（**仅**当规则文本明确同一模块/同一机制、且不与本次方案冲突时可作辅证，**不能单独作为定论**）。**只要任一来源能由 L1/KB 明确规则/已确认评论直接支撑，就不要列入待确认**，而是作为已知规则写入功能清单/方案切面并标 `[来源: ...]`。只有这些来源全部查无据、或答案依赖需登录的原型图（mastergo/lanhu）、或必须由产品拍板的业务决策时，才列入待确认，并在“已检索”里写明已查来源与 JQL/命中数。**不要把能从已加载证据确认的点抛给人工**。

没有待确认问题时写 `无`。

### 1.5 测试范围

必须分成：

- 覆盖范围：只列 L1 切面和必要边界。
- 不覆盖范围：明确写权限、导出、排序、列自定义、修改记录、提示文案等方案未提维度是否不覆盖。

### 1.6 测试点数量预估

输出：

- 总测试点数预估：N
- 优先级分布预估：P1 a / P2 b / P3 c
- 说明：为什么这个数量符合必要充分原则

---

## 2. 同时生成 `questions.md`

如果 `analysis.md ## 4. 待确认问题清单` 不为 `无`，必须生成 `questions.md`，严格按 `_kb/_global/markdown-artifact-schema.md §8` 写入。

每个问题必须包含：

- `## QN: <问题陈述>`
- `**问题**`
- `**来源**`（**逐条列出真实、可核对、能按图索骥找到的来源**，每条单独成句、必须带原句（加引号）并点名出处，按下列格式写：
  - 本工单方案：`本工单修改方案：“<原句>”`
  - 本工单评论：`本工单 <评论人> <日期> 评论：“<原句>”`
  - 关联工单评论/方案：`<工单号> <评论人> <日期> 评论：“<原句>”`
  - 知识库规则：`业务规则 <§章节>：“<原句>”`
  **引用评论/讨论必须点名「工单号 + 评论人 + 日期 + 原句」**，不能只说“提到了什么”。
  **严禁**：① 搜索过程——JQL 语句、`已检索/已查/未命中/未找到`、`§`、文件名（如 `requirement.md`）（JQL 清单只进 `analysis.md` 的「已检索」，绝不进 `questions.md`）；② **无主语、无出处的笼统标签**——`背景信息`、`关联讨论`、`相关讨论`、`有提到`、`资料显示`，以及单独出现而不带「工单号/评论人/日期/原句」的 `需求方案`、`评论`、`关联工单`；③ **L3 背景不是依据**——`客户与背景`、`背景与价值`、`客户场景`、`期望目标`、`功能详情`（requirement.md §2 等）只供理解上下文，禁止作为来源。
  反例（不合格，找不到出处）：`关联讨论提到“支持传入条件：客户代码”`。
  正例（合格，可溯源）：`EAR-252916 陈嘉豪 2026-06-03 评论：“接口功能为查询批发客户信息，支持传入条件‘客户代码’”`。
  若确实查无可确认来源，本字段写 `无可确认来源，待产品/原型确认`，**不要堆砌搜索过程或笼统标签冒充依据**。）
- `**可能场景**`
- `**影响范围**`
- `**✅ 答案**`

`✅ 答案` 下方只放 HTML 注释占位，不要替用户填写。用户明确回答后才写入答案。

如果没有待确认问题：

- 不生成新的 `questions.md`；或如果旧文件已存在，覆盖为：

```markdown
# 待确认问题清单 — <ticket>

无
```

---

## 文件输出规则

- `analysis.md` 和 `questions.md` 必须遵循 `_kb/_global/markdown-artifact-schema.md`。
- 完成后报告只输出在聊天回复中，不写入 Markdown 产物。
- 禁止在文件正文写 `✓ 已分析`、`请回答 Q1`、`下一步运行 /qa:points` 等聊天式提示。
- 禁止使用 `<b>`、`<strong>`、损坏 span、HTML 实体转义。

---

## 完成后聊天报告

回复用户时只简述：

```text
已生成 analysis.md；待确认问题 K 个，questions.md 已生成/无需生成；总体风险=高/中/低。
下一步先运行 /qa:resolve（自动消解有据待确认 + 漏问审查）；之后只回答 questions.md 中仍标「需人工」的问题，再运行 /qa:points。
```
