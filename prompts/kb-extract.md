# /qa:kb-extract — 评审后知识沉淀

> 用法：`/qa:kb-extract`（在工单目录工作树中）
> 时机：周一用例评审 / 周二预发后，确认本次新发现的业务规则该入库

---

## AI 任务

### 1. 扫读当前工单产物
- `requirement.md ## 8. Q&A（来自 /qa:analyze 待确认问题）`
- `questions.md`（已回答的待确认问题——人工答案为终裁）
- `business-context.md` 中"§6 与历史规则的关系/潜在冲突"
- `_resolve.md`（证据消解记录，如有）/ `_spot-check.md`（强抽检确认问题，如有）
- `analysis.md ## 4. 待确认问题清单`（已澄清的）

> **禁止**从 `test-design.json` 正文反向提炼规则：JSON 是纯 CodeArts 用例树，不含待确认清单；规则来源只取上面的 markdown 工程产物。

### 2. 识别"应该入库"的新规则
判断标准：
- **跨工单可复用**——下次再做类似需求会用到
- **覆盖产品行为**——而非工单一次性细节
- **明确无歧义**——已澄清，不是 `[待确认]` 状态
- **不与现有 `rules.md` 重复**
- **必须源自 L1**（`CLAUDE.md §3.3 第 2 条`）：
  - 工单 description 中"修改方案 / 解决方案"段
  - 评论中产品作者明确写的"补充方案" / "方案确认" / "方案修订"
  - **禁止从以下来源提取规则**：
    - 客户场景 / 用户问题 / 业务背景（L3 — 仅作背景理解）
    - 评论中开发提的接口名、实施细节、被否决的早期方案（L3）
    - 评论中 QA / 开发的提问与讨论（L2 — 仅作风险/边界判断依据）
    - Jira 截图（不是 UI 真理来源）
  - **辨识方法**：每条候选规则必须能指向 `requirement.md §3 修改方案` 或评论中产品作者明确的方案段。若指向 §2 / §5 / §6 / §7 → **不入库**，仅在 `kb-proposal.md` 中标注"候选规则来源不可靠（L2/L3），建议人工评审决定是否升级为 L1 入库"。
- **"业务原因" / "业务场景" 注释**：可作为规则的 **Why 注释**附在规则正文后（用 `- 业务原因：...` / `- 业务场景：...` 起头），但**必须**有对应的 L1 规则正文先存在；不允许只有"业务原因"没有规则正文的孤立条目

### 3. 生成入库 Patch 提案
对每条新规则，输出：

```markdown
## 候选规则 #1

### 建议入库位置
`_kb/projects/<product>/rules.md` §X.Y `<段落标题>`
（如该章节不存在，建议新建章节）

### 规则内容（拟入库正文）
<拟写入的 markdown 段落>

### 来源
- `tickets/<product>/<...>/requirement.md` 第 N 段
- 用户在 /qa:analyze Q1 的回答（YYYY-MM-DD 对话）
- Jira EAR-xxxxxx 评审意见（来自 Confluence 或评论）

### 是否与现有规则冲突
- 无冲突 / 冲突如下：
  - 现有：`_kb/projects/<product>/rules.md §A.B` 写明 "..."
  - 本次：写明 "..."
  - 建议：保留两者并加注变更记录 / 替换 / 其他

### 入库后影响
- 哪些历史工单的用例需要复审？（如 EAR-xxx 涉及同模块）
- 是否需要更新 `terms.md` / `modules.md`？

### 操作（用户确认后）
- [ ] 写入 `rules.md`
- [ ] 在 `rules.md` 文件末尾"变更记录"表追加一行
- [ ] 同步更新 `terms.md`（如有新术语）
```

### 4. 同时识别"应该入库"的术语
- 如本次新出现的业务术语在 `terms.md` 中没有 → 也列入提案

### 5. 同时识别"应该入库"的反例
- 如评审中 AI 写错了某种用例（典型反例）→ 提议追加到 `case-samples/style-notes.md` §9（反例与修正）

---

## 输出

整体输出一份**提案文档**，写到：

```
tickets/<product>/<sprint-date>/<ticket>/kb-proposal.md
```

包含：
- N 条候选规则
- M 条候选术语
- K 条候选反例

末尾给一个总结：

```
建议入库总数：N + M + K

人工动作：
1. 逐条审：保留 / 修改 / 放弃
2. 保留的部分，用户说"应用全部" / "应用 #1, #3" → AI 自动写入对应文件
3. 我自动给 rules.md 追加变更记录条目
```

### 写入流程（用户审批后）

用户回复"应用全部"或"应用 #X, #Y" → AI 执行：
1. **追加**到对应文件的对应章节（**不要重写整个文件**）
2. 在 `rules.md` / `terms.md` 文件末尾的"变更记录"表加一行：

```markdown
| 日期 | 工单 | 改动 |
|---|---|---|
| YYYY-MM-DD | EAR-xxxxxx | §X.Y 新增 / §A.B 修订 |
```

3. **同步更新 `rules.md §0 目录索引`（硬规则）**：
   - 新增主章节（`## N. 标题`）时：必须在 `§0` 对应业务域分组里追加一行 `§N 章节标题` 引用
   - 新增小节（`### N.X`）只在补充正文，不必动 §0（除非该小节代表新业务子域）
   - 新增"仅 XXXX 启用"专题：必须同步更新 `§0.16 客户定制专题速查表`
   - 若同一业务域已聚集多个相关章节（如 §6/§16/§19/§220 都是拣货）：在新章节标题用"（与 §X 互补，YYYY 累积）"显式交叉引用
   - **不要**把新章节随便放最后——按业务域归属插入对应分组（即使章节号继续递增也要在 §0 索引里归到正确域下）

4. **业务模块变更允许整体重排 §0**：
   - 当本次需求新增**一级业务大模块**（如"前置仓 WMS"作为新形态）、或多个章节随产品改版**重命名**时
   - 允许重新规划 §0 目录结构（新增分组 / 拆分分组 / 重命名分组标题）
   - 重排前先输出 §0 变更对照表（旧分组→新分组），用户确认后再写

5. **`modules.md` 与 `terms.md` 联动检查**：
   - 新规则涉及**新模块路径** → 同步把模块名加入 `modules.md` 对应平台的模块树，并标注 `→ rules.md §N`
   - 新规则引入**新术语 / 新平台前缀 / 新业务模式** → 同步追加到 `terms.md` 对应段
   - 新规则涉及**新客户定制专题** → 必须同时更新 `rules.md §0.16` 和 `modules.md §6.3 客户定制专题命名规则`

6. **写入后必跑 `scripts/kb-check-toc.py` 强制校验（阻断式）**：

   ```bash
   python scripts/kb-check-toc.py --product <product>
   ```

   - **退出码 0**（"✅ §0 同步完整"）→ 才可宣告入库完成
   - **退出码 1**（"❌ 游离章节" / "❌ 死引用"）→ **禁止**宣告完成；按报告输出补全 §0 索引后**重跑**，直至退出码 0
   - 如脚本报告"§0.16 客户定制速查表可能漏列 N 处"（提示级，非阻断）→ 评估是否为整章定制；若是必须补表，若仅节级可忽略
   - 严格模式（连节级定制都校验）：加 `--strict-custom`，但默认不强制——节级定制由 §0.16 表只覆盖整章

   **AI 行为约定**：执行 `/qa:kb-extract` 写入步骤的**最后一个动作**必须是跑该脚本并显示输出；不允许跳过、不允许仅在文字里声称"已检查"。

---

## 抗幻觉硬规则

1. **只提议有依据的规则**：必须能在 requirement.md / business-context.md / 用户回答中找到出处
2. **不要美化措辞**：保留用户原话或需求原文
3. **冲突先提示，不擅自覆盖**：让用户拍板
4. **不写未澄清的内容**：仍标 `[待确认]` 的，**不入库**
